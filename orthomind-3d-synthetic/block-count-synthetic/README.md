## Block Counting Data Generation and Three-View Descriptions

This directory provides a complete pipeline to **generate block layouts → render images → build VLM training/evaluation datasets → generate three-view JSON descriptions**.

### 1. Layout generation: `Block_gen.py` / `Block_gen_large.py`

- **`Block_gen.py` (small grids, exact enumeration)**
  - Targets small grids such as 2×2×2 and 3×3×3 and exhaustively enumerates all layouts that satisfy the "unique three-view solution" constraint, using traversal with pruning.
  - Suitable when you need full coverage of all solutions in small grids.
  - Output: `{output_dir}/levelRCH/block_solutions.json`, e.g. `OUTPUT_DIR/level333/block_solutions.json`.

```bash
python Block_gen.py --R 3 --C 3 --H 3 --output-dir OUTPUT_DIR
```

- **`Block_gen_large.py` (large grids, random search)**
  - Targets large grids such as 4×4×4 and 5×5×5 and uses a Min-Conflicts search to randomly find valid layouts.
  - Suitable when you want many large-grid samples without enumerating all solutions.
  - Output: `{output_dir}/levelRCH/block_solutions.json`.
- Example (generate 100 samples of 3×3×3 with the large generator):

```bash
python Block_gen_large.py --R 3 --C 3 --H 3 --num 100 --output-dir OUTPUT_DIR
```

### 2. Rendering images and matrices: `render_block.py`

- **Input**: `block_solutions.json` (generated by `Block_gen_large.py`).
- **Output (one subfolder per case)**:
  - Three orthographic views: `3view-top.png`, `3view-front.png`, `3view-left.png`;
  - Isometric views: `left-view-45.png`, `right-view-45.png` (mainly used for VLM questions);
  - Height matrix: `matrix.json` (2D height map `h[x][y]`);
  - Color matrix (optional if random colors are enabled): `color.json` (`[y][x][z] -> color_id`);
  - Answer: `answer.txt` (total number of blocks).
- Example:

```bash
python render_block.py --file OUTPUT_DIR/level333/block_solutions.json --num-cases 1000
```

### 3. Three-view JSON descriptions: `build_cube_views_json.py`

- **Purpose**: Before building train/eval sets, generate a **text-form three-view description** for each sample so models can reason about block counting from JSON.
- **Input**: Output directory of `render_block.py` (contains `level***` subfolders; each case has `matrix.json` and optional `color.json`).
- **Output (inside each case folder)**: `view_desc.json`
  - `shape`: `{C, R, H}` grid size;
  - `views.left / front / top`:
    - Uses an **orthographic three-view coordinate system**;
    - For each "ray" (fixed `(y,z)`, `(x,z)`, or `(x,y)`), keep **only the nearest truly visible block under the isometric view**;
    - If color exists, include `color_id` and `color_name`.
- Visibility is computed by reusing `utils.generate_voxel_matrix` (consistent with the `left-view-45.png` isometric view).
- Example (color data):

```bash
python build_cube_views_json.py --input-dir OUTPUT_COLOR
```

### 4. Build the standard block-counting dataset: `build_cube_counting_dataset.py`

- **Purpose**: Build a **color-agnostic block-counting** VLM dataset from rendered results (e.g. `OUTPUT_DIR`).
- **Data sources** (per case):
  - `left-view-45.png` as the question image;
  - `answer.txt` as the total block count.
- **Output**:
  - `data/eval/cube_counting/cube_counting.jsonl` + `images/`;
  - (optional) `data/train/cube_counting/cube_counting.jsonl` + `images/`.
- Supports:
  - `--eval-num`: split eval/train by total count;
  - `--eval-per-level`: control eval size per level.
- Example:

```bash
python build_cube_counting_dataset.py \
  --eval-per-level "222:20,333:180,444:150,555:150" \
  --input-dir OUTPUT_DIR
```

### 5. Build the color block-counting dataset: `build_cube_counting_color_dataset.py`

- **Purpose**: Use color information to generate multiple color-specific questions per image (e.g. "How many red blocks are there?").
- **Data sources**:
  - `matrix.json` + `color.json` + `left-view-45.png`;
  - Visibility computed by `utils.generate_voxel_matrix`;
  - `COLOR_ID_TO_NAME` maps color IDs to names (black/white/red/...).
- **Output**:
  - `data/eval/cube_counting_color/cube_counting.jsonl` + `images/`;
  - (optional) `data/train/cube_counting_color/cube_counting.jsonl` + `images/`.
- Example question template:
  - `"How many red blocks are there in the picture? Return your final response within \\boxed{}."`

### 6. Example pipeline: `generate_full_data.sh`

- A simple script showing the command order for **layout generation → rendering → dataset construction**.
- Includes two command sets (no color / with color), mostly commented and ready to enable:
  1. Call `Block_gen_large.py` to generate `block_solutions.json` for each level;
  2. Call `render_block.py` to render images and matrices;
  3. Call `build_cube_counting_dataset.py` to build the standard dataset.

Use it as a template to adjust parameters for your data scale and levels, and after rendering run:

```bash
python build_cube_views_json.py --input-dir OUTPUT_DIR
python build_cube_views_json.py --input-dir OUTPUT_COLOR
```

This generates three-view JSON descriptions for all samples, which can be used for downstream VLM training with chain-of-thought.
